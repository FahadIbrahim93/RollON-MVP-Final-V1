## High-Priority Issues (P0/P1)

1. **P0 — Test signal integrity regression existed**
   - Vitest ran tests under `node_modules` because default excludes were overridden.
   - Impact: flaky/false negatives, wasted CI time, reduced trust in test pipeline.
   - Status: **fixed in this audit**.

2. **P1 — Silent remote→mock fallback policy in API layer**
   - Current behavior can suppress upstream API/auth failures and continue with local dataset.
   - Impact: users may see stale/inconsistent data; operational incidents become harder to detect.
   - Recommendation: gate fallback by explicit environment + user role + feature flag + visible degraded-mode UI.

3. **P1 — Coverage quality mismatch in risk-heavy modules**
   - Coverage is above threshold overall, but `api.ts` and several page/store branches remain weak.
   - Impact: high-risk integration logic is not proportionally protected.
   - Recommendation: prioritize branch/contract tests around auth token parsing, network error mapping, and mutation flows.

4. **P1 — Security scan unreliability in CI-like environment**
   - `npm audit` blocked by registry endpoint restrictions in this execution context.
   - Impact: vulnerability posture cannot be claimed clean from this run.
   - Recommendation: add deterministic secondary scanner (e.g., `osv-scanner`/Snyk/GitHub Dependabot) and archive machine-readable reports.

5. **P1 — Performance governance still soft**
   - Large emitted chunks indicate potential runtime costs at scale.
   - Impact: degraded first-load/TTI for constrained devices.
   - Recommendation: enforce bundle budget checks with fail-fast thresholds per route/chunk.

## Technical Debt Register (Top items)

- API client currently owns transport, fallback policy, and local-store orchestration simultaneously.
- Inconsistent strictness for error semantics (`throw Error(status)` pattern is too coarse for UX and observability).
- Testing strategy is still heavily unit-centric; deeper integration checks for async user journeys are limited.
- Security controls rely heavily on runtime conventions; stronger schema validation and response hardening should be standardized at boundaries.

## Concrete Improvement Plan (best-next execution order)

### Phase A (Immediate, 1–2 days)
1. Keep the Vitest exclude fix and add a regression test/assertion on config invariants.
2. Refactor `api.ts` fallback behavior:
   - no silent fallback for authenticated endpoints;
   - explicit degraded mode for read-only public endpoints only;
   - structured error taxonomy.
3. Add targeted tests for `api.ts` error/fallback branches and auth token parsing edge cases.

### Phase B (Short-term hardening, 3–5 days)
4. Introduce route-level performance budgets and CI enforcement.
5. Add automated accessibility check in CI (`axe`/Lighthouse budget with failing thresholds).
6. Add security scan redundancy with at least two independent sources (registry audit + OSV/Dependabot).

### Phase C (Production excellence, 1–2 weeks)
7. Add observability SLOs: error-rate alerting, degraded-mode telemetry, and release health dashboard.
8. Document rollback playbook and perform one game-day rollback rehearsal.
9. Move API transport to dedicated adapter + domain services for cleaner module boundaries.
